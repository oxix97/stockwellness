# 📑 stockwellness-batch 종합 코드 리뷰 리포트

## 1. 총평 (Overall Architecture)
본 프로젝트는 **이벤트 기반 아키텍처(Kafka)**를 도입하여 배치와 API 모듈 간의 결합도를 성공적으로 낮추었으며, 헥사고날 아키텍처의 원칙을 충실히 따르고 있습니다. 특히 **조건부 이벤트 발행** 설계는 운영 효율성 측면에서 매우 뛰어납니다. 다만, 멀티스레드 환경에서의 **데이터 안전성**과 **대량 처리 성능** 부분에서 즉각적인 보완이 필요합니다.

---

## 2. [Urgent] 즉시 수정 권고 사항 (Bugs & Safety)

### 2.1 스레드 안전성 확보 (`@JobScope` 추가)
*   **문제**: `StockPriceSyncEventListener`가 싱글톤으로 관리되어, 여러 배치가 동시에 실행될 경우 업데이트된 종목 리스트(`updatedSymbols`)가 섞이거나 유실될 수 있습니다.
*   **해결**: 해당 컴포넌트에 `@JobScope`를 추가하여 각 배치 실행마다 독립적인 인스턴스를 생성해야 합니다.

### 2.2 스케줄러 실행 결과 검증
*   **문제**: `StockwellnessScheduler`가 앞선 단계의 성공 여부를 확인하지 않고 다음 단계(Price -> Sector)를 실행합니다.
*   **해결**: `jobLauncher.run()`의 반환값인 `JobExecution`의 상태를 체크하여, `FAILED`인 경우 후속 작업을 중단하는 로직을 추가해야 합니다.

---

## 3. [Performance] 성능 최적화 사항

### 3.1 Redis 캐시 무효화 방식 개선
*   **문제**: `StockPriceUpdateConsumer`에서 수천 개의 종목을 루프를 돌며 개별적으로 `evict` 호출 시 네트워크 오버헤드가 발생합니다.
*   **해결**: Redis Pipelining을 사용하거나, 특정 패턴의 키를 한꺼번에 삭제하는 벌크 처리 방식으로 전환을 고려해야 합니다.

### 3.2 비동기 스레드 풀 관리
*   **문제**: `BatchAdminController`에서 공용 `ForkJoinPool`을 사용하여 배치를 실행하고 있습니다.
*   **해결**: 기존에 정의된 `kisBatchExecutor` 등 전용 `TaskExecutor`를 사용하도록 명시하여 시스템 자원 간섭을 방지해야 합니다.

---

## 4. [Maintainability] 유지보수성 및 운영 개선

### 4.1 로그 표준화 (Observability)
*   **제언**: `MDC`를 활용하여 로그에 `JobInstanceId`를 포함시키십시오. 여러 배치가 동시에 돌 때 특정 작업의 흐름만 추적하기가 훨씬 수월해집니다.

### 4.2 API 응답 피드백 강화
*   **제언**: 배치 실행 API 호출 시 단순 성공 메시지 대신, 배치의 상태를 모니터링할 수 있는 `ExecutionId`나 상태 조회 URL을 응답에 포함하십시오.
