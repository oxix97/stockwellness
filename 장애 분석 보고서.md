### 3. 원인 분석

이번 장애는 캐시 계층에서 2단계로 발생한 것으로 판단됩니다.

#### 1단계. 캐시 키 평가 실패

- `key = "#sectorCode"` 사용 시 파라미터 이름 메타데이터가 유지되지 않으면 SpEL이 값을 찾지 못함
- 그 결과 null key 예외가 발생하며 API 500으로 이어짐

#### 2단계. 캐시 역직렬화 실패

- 이후 Redis에서 캐시 값을 읽는 과정에서 JSON 역직렬화 예외 발생
- 이는 기존 Redis 데이터가 현재 serializer와 다른 형식으로 저장되었거나, 캐시 객체 구조 변경으로 인해 호환되지 않기 때문으로 추정됨

즉, 이번 장애의 본질은  
**“Spring Cache key 설정 취약점 + Redis serializer/기존 캐시 데이터 호환성 문제”**로 정리할 수 있습니다.

### 4. 즉시 조치안

#### 4-1. 코드 수정

- 기존:

@Cacheable(value = "sectorComparison", key = "#sectorCode")

- 수정:

@Cacheable(value = "sectorComparison", key = "#p0", unless = "#result == null")

#### 4-2. 입력값 방어

- `sectorCode`가 null/blank인 경우 예외 처리 또는 캐시 미적용 처리

#### 4-3. Redis 캐시 정리

- `sectorComparison` 관련 기존 캐시 키 삭제
- serializer 변경 이력이 있다면 배포 시 캐시 초기화 또는 cache versioning 적용

---

### 5. 근본 개선안

#### 5-1. 캐시 키 전략 표준화

- 파라미터명 기반 SpEL 대신 `#p0`, `#p1` 등 인덱스 기반 사용
- null/blank 입력은 `condition`으로 캐시 제외

#### 5-2. 캐시 버전 관리

- DTO 구조 또는 serializer 변경 시 cache name에 버전 부여
- 예: `sectorComparison:v2`

#### 5-3. 배포 체크리스트 추가

- serializer 변경 여부 확인
- 캐시 대상 클래스 구조 변경 여부 확인
- 기존 Redis 데이터와의 호환성 검증
- 스테이징에서 기존 캐시 유지 상태로 테스트

#### 5-4. 관측성 강화

- cache miss/hit 로그
- serializer 예외 전용 알림
- cache namespace별 에러 건수 모니터링
